Cloud Config Right Sizing Agent

Technical Reference & Guide

This document covers how to use Gemini Enterprise Agent Designer to build the SOW-to-config agent, how to approach prompt engineering, and how the supporting tooling (sanitization, validation, scoring) works.

Agent Designer - Step-by-Step Guide

Getting Started

  1. Open Gemini Enterprise
  2. In the left sidebar, click Agents
  3. Click + New agent
  4. The Agent Designer page appears with a chat box at the bottom

Two Ways to Build

MethodBest for
Chat pane - Describe the agent in plain languageGetting started quickly
Designer pane - Edit instructions, model settings, data sources directlyFine-tuning after the initial version

Step 1: Create the Agent

Paste the following into the chat box:

Create an agent that accepts a Cadence Managed Cloud Statement of Work
(SOW) document and generates a chamber configuration JSON.

The agent should extract:
1. Platform provider (AWS, GCP, or Azure) and chamber location
2. Total number of users and their account locations
3. Hardware: interactive servers (count, instance type, vCPUs, RAM),
   storage (backup GB, scratch GB)
4. Software/licensed materials: each Cadence product name and quantity
5. Additional services: PDKs to install
6. Output only valid JSON  - no markdown, no explanations
7. Set customer_name to "[REDACTED]"

Step 2: Test the Agent

  1. In the agent preview, type: "Here is a SOW document" and attach a sample SOW PDF
  2. Review the JSON output - Is it valid? Are all fields present? Do the values make sense?
  3. Note what the agent gets right and what it misses

Step 3: Open Agent Designer for Fine-Tuning

Click "Open in Agent Designer" to access the full configuration. The Instructions field is where prompt engineering happens.

Prompt Engineering Guide

Principle 1: Be Specific, Not Vague

Bad: "Generate a cloud config from the SOW."

Good: "Read the SOW document and generate a JSON configuration with the following structure. Extract the values directly from the document. If a value is not explicitly stated, infer it conservatively based on the context."

Principle 2: Include the Schema in the Instructions

The agent needs to know exactly what the output should look like. Include the full JSON schema or a complete example of the expected output structure directly in the instructions.

Principle 3: Tell the Agent What NOT to Do

- Do not include markdown code fences around the JSON
- Do not add explanations or commentary before or after the JSON
- Do not include software products not listed in the SOW
- Do not include customer addresses, contact info, signatures, terms, or payment info
- Do not guess pricing or budget information

Principle 4: Set Clear Rules for Ambiguous Cases

- Use exact field names: "product_name" (not "name"), "instance_type", "vcpus", "memory_gb"
- For software licenses, remove trademark symbols like (R) and (TM) from product names
- Always set customer_name to "[REDACTED]"
- If backup or scratch storage is not mentioned, omit those fields
- Only include PDKs that are explicitly listed in the SOW

Few-Shot Prompting - How to Use Reference Examples

What is Few-Shot Prompting?

Instead of just giving the agent instructions, show it examples of correct input->output pairs before asking it to process a new SOW. This dramatically improves accuracy because the agent learns the expected mapping pattern from real examples.

How to Add Examples to the Agent Instructions

REFERENCE EXAMPLES:

Example 1  - Small chamber (10 users):
Input SOW excerpt:
"Platform Provider: AWS, Chamber Location: AWS Frankfurt.
10 users (5 Berlin, 5 Istanbul). 1 interactive server,
M6i.4xlarge, 16 vCPUs, 64 GB RAM. 1000 GB backup storage.
3 Virtuoso Schematic Editor L, 9 Spectre MMSIM.
2 customer-provided PDKs."

Expected output:
{
  "customer_name": "[REDACTED]",
  "platform_provider": "AWS",
  "chamber_location": "AWS Frankfurt",
  "total_users": 10,
  "user_accounts": [
    {"count": 5, "location": "Berlin Germany"},
    {"count": 5, "location": "Istanbul Turkiye"}
  ],
  "hardware": {
    "interactive_servers": [
      {"count": 1, "instance_type": "M6i.4xlarge", "vcpus": 16, "memory_gb": 64}
    ],
    "storage": {"backup_storage_gb": 1000, "scratch_storage_gb": 1000}
  },
  "software_licenses": [
    {"product_name": "Virtuoso Schematic Editor L", "quantity": 3},
    {"product_name": "Spectre MMSIM with Spectre X Simulator", "quantity": 9}
  ]
}

Tips for Effective Few-Shot Examples

  • Include at least 2-3 examples covering different chamber sizes (small team, large deployment)
  • Match the exact schema - examples must use the same JSON field names as the expected output
  • Use realistic content - the more examples look like real Cadence SOWs, the better the agent performs
  • Cover variation - include examples with different instance types, different numbers of licenses, different regions
  • Use real SOW/config pairs (with PII removed) when available

Uploading Reference Files Instead

Instead of pasting everything into the instructions, reference materials can be uploaded as files in Agent Designer:

  1. In Agent Designer, look for Add files or Data sources
  2. Upload SOW/config pairs as reference documents
  3. Update the instructions: "Use the uploaded reference examples to understand the expected mapping pattern"

The Iteration Loop

After each test run, follow this cycle to improve the agent:

  1. Run a SOW through the agent
  2. Compare the output to what was expected
  3. Identify the gap - missing field? wrong value? extra text?
  4. Update the instructions to address that specific gap
  5. Re-test the same SOW to verify the fix
  6. Cross-test a different SOW to make sure the fix didn't break something else

Common Issues and Fixes

ProblemSolution
Missing fields in the JSONAdd the missing fields to the schema in the instructions and specify they are required
Wrong values (e.g., wrong instance type)Add a rule explaining how to choose that value, or add a reference example that covers that case
Extra text around the JSONAdd "Output ONLY valid JSON. No markdown, no explanations, no code fences"
Inconsistent output formatAdd a complete example of the exact expected JSON structure
Agent ignores part of the SOWAdd "Extract ALL relevant information" and specify which sections to look for
Agent hallucinates services not in the SOWAdd "Only include services explicitly mentioned in the SOW"

Target JSON Schema

The agent should generate JSON matching this structure. This may be updated on dev day based on the actual Terraform configuration format.

{
  "customer_name": "[REDACTED]",
  "platform_provider": "AWS | GCP | Azure (required)",
  "chamber_location": "string, e.g. AWS Frankfurt (required)",
  "third_party_licence_chamber_required": "boolean",
  "total_users": "integer (required)",
  "user_accounts": [
    { "count": "integer", "location": "string, e.g. Berlin Germany" }
  ],
  "hardware": {
    "interactive_servers": [
      {
        "count": "integer",
        "purpose": "string, e.g. interactive server",
        "instance_type": "string, e.g. M6i.4xlarge",
        "vcpus": "integer",
        "memory_gb": "integer"
      }
    ],
    "storage": {
      "backup_storage_gb": "integer",
      "scratch_storage_gb": "integer"
    }
  },
  "software_licenses": [
    { "product_name": "string, e.g. Virtuoso Schematic Editor L", "quantity": "integer" }
  ],
  "additional_services": {
    "pdks_to_install": [
      { "name": "string", "source": "customer_provided | cadence_provided" }
    ]
  }
}

Supporting Tools - Sanitize, Validate & Score

Sanitize - PII Redaction

Customer SOWs contain sensitive information. The sanitization pipeline extracts text from the PDF and redacts PII before any content reaches the model.

What gets redactedExampleReplacement
Email addressesjohn@acme.com[REDACTED_EMAIL]
Phone numbers(555) 123-4567[REDACTED_PHONE]
Street addresses123 Main Street[REDACTED_ADDRESS]
Account/contract IDsAccount ID: ACME-001[REDACTED_ACCOUNT_ID]
Client/customer namesClient Name: Acme CorpClient Name: [REDACTED]
Pricing/budget linesBudget: $45,000/month[REDACTED_PRICING]
SOW headings/order numbersOrder #C-GNX01[REDACTED_ORDER_ID]
Signatures, terms, footer textCustomer Signature: ___[REDACTED_SIGNATURE]
What stays intact: Platform provider, chamber location, hardware specs, storage amounts, software license names and quantities, user counts and locations, PDK requirements.

Validate & Score - Output Quality

Validation checks the generated JSON against the target schema - are all required fields present, are data types correct, are enum values valid.

Scoring compares the generated config to a known-good config field by field and reports a percentage accuracy.

CategoryFields Checked
Platformplatform_provider, chamber_location, third_party_licence_chamber_required, total_users
Hardwareinteractive_servers[0].count, instance_type, vcpus, memory_gb
Storagebackup_storage_gb, scratch_storage_gb
SoftwareEach license: product_name and quantity
Usersuser_accounts (count + location per entry)
Validation = "Is the JSON structurally correct?" (yes/no gate)
Scoring = "Are the values accurate compared to the expected answer?" (percentage)
